The `admin_*` policies on `config_assets`/`pm_schedule`/`cmc_contracts` were decoded on 2026-05-09 (blocker #1 resolved). All three tables use the **same write-gate expression**:

```sql
-- config_assets / pm_schedule / cmc_contracts
-- For: admin_write_*, admin_update_*, admin_delete_*
qual / with_check:
  ((auth.jwt() -> 'app_metadata'::text) ->> 'role'::text) = 'admin'

-- For: auth_read_*
qual:
  true   -- any authenticated user can read
```

**Plain-language meaning:** "admin" today = a Supabase auth user whose JWT `app_metadata.role` claim equals the literal string `'admin'`. Nothing else qualifies — not `user_metadata`, not email allowlist, not a DB table.

**Practical consequences:**
- `a***@3imedtech.com` has `app_meta_role='admin'` → **DB-side writes succeed.**
- `m***@3imedtech.com` has `app_meta_role='viewer'` → **DB-side writes are denied.** The Manager UX exists today only because the client-side `canManagePM()` email allowlist treats this user as a manager — but PostgREST silently rejects any actual write attempt from this session today (the app generally doesn't exercise those write paths because the only Manager-driven write surface is `dbSavePM()` upserts, which themselves are gated by `admin_write_pm_schedule` and would also fail).
- `e***@3imedtech.com` has `app_meta_role='viewer'` → DB-side writes denied. Matches client expectations (Engineer is read-only).

**Implication for v1.4.1:** the existing `admin_*` policies must remain untouched in Phase 1. They give the Admin user a working write path today; replacing them prematurely would break production. The `user_roles` foundation in 0003 is additive — it does not modify any existing policy. Phase 2 then transitions writes to `app_can_write()` once `user_roles` is seeded and verified. See §3.7 Phased RLS Transition below.

### 3.5 `user_roles` Design (Option B Detail)

**Table:**
```sql
create table public.user_roles (
  user_id     uuid          primary key references auth.users(id) on delete cascade,
  email       text          not null,
  role        text          not null check (role in ('admin','manager','viewer')),
  active      boolean       not null default true,
  created_at  timestamptz   not null default now(),
  created_by  uuid          references auth.users(id),
  updated_at  timestamptz   not null default now(),
  updated_by  uuid          references auth.users(id),
  note        text
);
```
- **PK = `user_id`** so the row tracks the auth identity (FK with cascade-on-delete keeps it in sync).
- **`email` is denormalized** for grep-ability in admin UI; not the identity.
- **`role` constrained** to `admin` / `manager` / `viewer`. No separate `superadmin` value — Audit Log access stays gated by the existing email check (unchanged in v1.4.1).
- **`active` boolean** for soft disable. Preserves history.

**Helper functions** (all `stable security definer set search_path = public`):
- `public.app_user_role() returns text` — returns the caller's role string or null.
- `public.app_can_write() returns boolean` — true for admin or manager.
- `public.app_is_admin() returns boolean` — true for admin only.

`security definer` is essential: helpers query `user_roles` from inside RLS policies on `user_roles` itself; without `definer` they would recurse. Locked `search_path = public` prevents schema-injection attacks.

**App integration (Phase 2 — separate review):**
- After login: `const { data } = await _sb.rpc('app_user_role'); _userRole = data || 'viewer';`
- `canManagePM()` becomes `_userRole === 'admin' || _userRole === 'manager'` — drop the email allowlist branch.
- Audit Log gate stays untouched (still email-based per current behavior).

| Current concept | DB representation |

| Audit Log access | Email-gated (unchanged, separate from `user_roles`) |

### 3.6 Last-Admin Lock-Out Guard

The user_roles write path is doubly guarded so the system cannot lose its last active admin:

**Layer 1 — `with check` clause on `v141_user_roles_write_admin` policy:**
```sql
with check (
  public.app_is_admin()
  and (
    (role = 'admin' and active = true)
    or exists (
      select 1 from public.user_roles ur
       where ur.role = 'admin'
         and ur.active = true
         and ur.user_id <> user_roles.user_id
    )
  )
)
```
- Allows: setting any user to admin+active; or modifying any user as long as ANOTHER active admin exists.
- Denies: demoting the last active admin (role flip) or deactivating them (`active=false`) when no other admin exists.

**Layer 2 — `BEFORE DELETE` trigger `trg_user_roles_block_last_admin`:**
PostgreSQL does not evaluate `WITH CHECK` on DELETE, so a separate `BEFORE DELETE` trigger raises `check_violation` if the row being deleted is the last `role='admin' AND active=true` row. The function is `security definer` so it can read user_roles without re-entering RLS.

**What this protects against:**
- Admin manually demoting themselves and losing access
- Admin accidentally deactivating their own row
- Admin running `delete from user_roles where role='admin'` as a sweep
- A misconfigured backfill ending with zero admins (caught by the §1d backfill block's `do $$ ... raise exception ... $$;` sanity check)

**What this does NOT protect against:**
- Service-role direct SQL (bypasses RLS by design — emergency unlock path)
- Admin removing the FK referenced row in `auth.users` (would cascade-delete the user_roles row; in that case, escalate to Supabase Auth dashboard)

**Test plan for the guard** (subset of §11):
- As admin: try `update user_roles set role='viewer' where user_id=auth.uid()` (self-demotion when sole admin) → must fail
- As admin: try `update user_roles set active=false where user_id=auth.uid()` (self-disable when sole admin) → must fail
- Add a second admin row; repeat both attempts on the original → must succeed
- As admin: `delete from user_roles where user_id=<sole admin>` → must fail with `check_violation`
- Service role: same delete → succeeds (intentional emergency path)

### 3.7 Phased RLS Transition (decoded admin_* policies → app_can_write())

The existing `admin_write_*` / `admin_update_*` / `admin_delete_*` policies on `config_assets`, `pm_schedule`, and `cmc_contracts` are **JWT app_metadata admin-only checks** (see §1 Existing RLS policy inventory for the decoded expression). Replacing them prematurely would lock the production Admin user out of the only working write path. v1.4.1 takes a **strict additive-then-migrate** approach.

#### Phase 1 (this migration, 0003) — additive only

- Creates `user_roles`, helpers, lifecycle tables.
- **Does NOT touch** any existing `admin_*` / `auth_read_*` policy on the legacy tables.
- The new `v141_lifecycle_*` policies on `asset_lifecycle` and `asset_lifecycle_history` are the only RLS policies that use `app_can_write()`.
- Result: Admin keeps writing today exactly as before (JWT-based). Manager-allowlist user is unaffected (still blocked at DB layer; Manager UX still client-only). No regression possible from a policy substitution.

#### Phase 1.5 (operational, post-migration on staging) — verify

After the migration is applied and `user_roles` is seeded:
- Sign in as Admin: `select public.app_can_write()` → `true` AND existing `admin_write_config_assets` still permits the write. Both gates align for this user.
- Sign in as Manager (the allowlist user): `select public.app_can_write()` → `true` (because `user_roles.role='manager'` was seeded) **but** `admin_write_config_assets` still returns `false` (because `app_meta_role='viewer'`). PostgREST will deny the write, surfacing the policy gap.
- Document this gap explicitly. Phase 2 closes it.

#### Phase 2 (separate migration + app deploy) — migrate

In a follow-on migration (target: `0004_*` or later), do **one** of these per table — **NOT both**:

- **Option B-Migrate:** drop the existing `admin_write_config_assets` etc. and create `v141_<table>_write_app_can_write` using `with check (public.app_can_write())`. Single source of truth; cleanest. Requires `user_roles` to be fully seeded and the app to be reading role via RPC.
- **Option B-Additive:** keep `admin_write_config_assets` AND add `v141_<table>_write_app_can_write` with `with check (public.app_can_write())`. PostgreSQL OR's permissive policies, so any user matching either gate can write. Lowest-risk; allows gradual cut-over. Use this if uncertainty about user_roles seed completeness.

Recommendation for the Phase 2 review: start with **Option B-Additive**. After 1–2 weeks of clean operation, run a follow-up migration to drop the legacy `admin_*` write policies (Option B-Migrate stage 2). Read policies (`auth_read_*` with `qual: true`) stay as-is — they don't need `app_user_role()` anything.

#### Strict prohibition (Phase 1 scope)

- **Do NOT replace `admin_*` policies in 0003.** Even with `create or replace policy ... if not exists` semantics, dropping one of the existing policies at this stage would create a window where Admin writes fail.
- **Do NOT seed Admin into `user_roles` and assume the legacy policy is now redundant.** The legacy policy and the new `app_can_write()` policy must both succeed for the Admin user's writes during the cut-over window.
- **Do NOT drop the existing `admin_*` policies in the rollback file** either. The rollback drops only `v141_*` policies and the helper functions; legacy policies are out of 0003's scope.

This is the safest sequence for production. It will not satisfy reviewers who want a single source of truth on Day 1 — that's a deliberate trade-off for zero-risk rollout.

## 5. Asset Code Immutability (Decided Policy #6)

**Decision:** asset_code is immutable after creation.

**DB enforcement:**
- `asset_lifecycle.asset_code` FK uses `on update restrict on delete restrict`. A SQL `update config_assets set code = 'NEW'` would fail if any lifecycle row references the old code. Code rename is forbidden at the DB layer.
- `config_assets.code` is already **PRIMARY KEY** (`config_assets_pkey`, confirmed by 2026-05-09 staging inspection). The 0003 §0 defensive unique-index block has been removed — uniqueness is already in place. Cross-table FK enforcement (with `on update restrict`) is sufficient to guarantee code immutability at the DB layer; a CHECK constraint or BEFORE UPDATE trigger preventing renames on `config_assets` itself is not added, because (a) the FK already blocks renames whenever lifecycle rows exist and (b) a per-row trigger would slow XLSX bulk upload.

**App enforcement:**
- Add Asset form: `code` field is editable (required) at create time.
- Edit Asset form: `code` field is **read-only** (rendered as plain text or disabled `<input>`).
- The future Edit form's submit handler must NOT include `code` in the upsert payload — even an unchanged value sent through `update` could be safe, but excluding it removes a class of accidental overwrite.

**Test plan:**
- After Phase 2 UI ships, attempt `update public.config_assets set code = 'X' where code = 'Y'` via SQL editor as service_role: must succeed only when no asset_lifecycle row references `Y`. If any reference exists, must fail with FK violation.
- App-side: open Edit Asset for a code with lifecycle history; the code field MUST render read-only. Attempt to bypass via DevTools (set `<input>` value, submit): the submit handler should not send `code` in the payload.

## 6. XLSX Bulk Upload — Decision: Option A (Upsert-by-Code)

**RESOLVED 2026-05-09.** XLSX bulk upload moves from `delete + insert` (full replace) to `upsert(deduped, {onConflict: 'code'})` before Add Asset UI ships. Companion to the master-source decision in §1 (DB is master).

### 6.1 Current behavior (the risk this resolves)

[`index.html:2635`](../index.html:2635) runs:
```js
const delAssets = await _sb.from('config_assets').delete().neq('code', '__sentinel__');
const { error } = await _sb.from('config_assets').insert(deduped);
```

This bulk `delete + insert` flow:
- Wipes EVERY row in `config_assets`, including app-created ones.
- Re-inserts only the rows in the uploaded XLSX.
- Loses `status`, `de_installed_at`, `created_by`, `note`, all v1.4.1 lifecycle fields.
- FK from `asset_lifecycle.asset_code` blocks `delete()` if any lifecycle rows exist (because we set `on delete restrict`). XLSX upload will silently fail or partially fail once Phase 1 DDL is applied and any lifecycle rows have been written.

### 6.2 Decision — Option A (Upsert-by-Code)

| Option | Behavior | Status |
|---|---|---|
| **A — Upsert by code** ✓ CHOSEN | Replace `delete + insert` with `upsert(rows, {onConflict: 'code'})`. Rows whose `code` exists in the DB are updated; rows whose `code` is new are inserted; DB rows whose `code` is missing from the XLSX are **NOT** deleted. | **Adopted.** |
| B — Preserve `app_created` rows during full replace | Two-class semantics. Rejected as overly complex. | Rejected. |
| C — Disable XLSX full replace once app CRUD is enabled | Largest behavioral change. Worth revisiting in v1.5+. | Deferred. |

### 6.3 Phase 2 implementation requirements (binding)

When the XLSX upload code is rewritten in Phase 2, it MUST:

1. **Upsert by `code`.** PostgREST: `_sb.from('config_assets').upsert(rows, { onConflict: 'code' })`. Replaces both lines at [index.html:2635-2637](../index.html:2635). The destructive `delete().neq('code','__sentinel__')` line is **removed**.
2. **Treat `code` as immutable PK.** XLSX cannot rename a code (changing `code` for an existing row would create a new row, not update the old). Match v1.4.1 product decision #6 and the FK `on update restrict` from `asset_lifecycle.asset_code`.
3. **Update only "XLSX-owned" fields on conflict** — name, model, town, state, channel, gradient, sw, compressor, coldhead, tesla, ast_id, alias_of. Do NOT include lifecycle fields (`status`, `de_installed_at`, `de_installed_by`, `created_by`, `updated_by`, `note`) in the upsert payload. PostgREST upsert with explicit column list achieves this; alternatively, omit those columns from the row objects passed in.
4. **Do NOT auto-delete DB rows missing from XLSX.** A row that's in `config_assets` but absent from the XLSX upload remains in `config_assets`. De-installation happens through the lifecycle/de-install workflow in the app, not by omission.
5. **Do NOT enable Manager XLSX permission in Phase 1.** The existing `admin_write_config_assets` policy (§1) still gates writes to `app_metadata.role='admin'`. Until §3.7 Phase 2 lands additive `app_can_write()` policies, **only Admin can run XLSX upload**. Phase 2 must explicitly decide whether Manager should be able to upload XLSX too — it's not a side effect of the upsert change.
6. **Pre-flight de-install audit.** Before each XLSX upload, the app should generate a diff: which DB codes are in the XLSX, which are not. The "not-in-XLSX" set is a candidate de-install list for admin review — but the app does NOT auto-de-install. Admin chooses which to de-install via the lifecycle workflow.

### 6.4 Hard requirement before Add Asset UI ships

**The Add Asset UI MUST NOT ship until the destructive `delete + insert` bulk replace is retired and the Option A upsert-by-code path is in place and verified on staging.** Specifically:

- [ ] Phase 2 commit replaces `delete + insert` at [index.html:2635](../index.html:2635) with upsert-by-code.
- [ ] Verified on staging: XLSX upload preserves an existing `app_created` row not in the XLSX file.
- [ ] Verified on staging: XLSX upload does not erase `status='de_installed'` flag from a row included in the upload.
- [ ] Phase 2 RLS migration (additive `v141_<table>_write_app_can_write` per §3.7) lands BEFORE or WITH the Add Asset UI commit.
- [ ] Visible UI confirmation that DB-is-master is the active model (e.g., a small "DB" indicator in Install Base panel header, or admin-only diagnostic showing DB row count vs INSTALL_BASE_V2 row count).

If any of these are not green, Add Asset UI does not ship. This is a hard-coded gate in the Phase 2 review.

## 9. Backfill Strategy

`asset_lifecycle` starts empty. We need historical contract rows from the existing PM_SCHEDULE and CMC_DATA tables to populate it so PM scheduling derived from `pm_required` and renewal urgency derived from `contract_end` keep working.

### 9.1 SQL exact-match phase (server-side, fast, reviewable)

Insert one `asset_lifecycle` row per (`config_assets`, source) pair where:
- exact match: normalized customer key (`lower(regexp_replace(...))`) AND normalized town match
- contract_type derived from PM/CMC `contract` field (`CMC` → 'cmc', `Warranty` → 'warranty', `Extended Warranty` → 'extended_warranty', else flag for review)
- contract_start / contract_end taken from source row
- `event = 'backfill'`, `source = 'sql:backfill_v1'`, `status = 'active'`
- skip if a row already exists for this asset_code

The exact-match SQL will be drafted in Phase 1.5 once schema is approved. It will be run in a transaction with an explicit `select count(*)` before COMMIT.

### 9.2 Node helper for fuzzy + town_model cases

Rows that don't exact-match get fed through the JS layered matcher (`_findContractForAsset` exact → fuzzy → town_model) running in a Node script that connects via the service role and writes the chosen lifecycle row + a history event with `source = 'node:backfill_fuzzy'`.

### 9.3 Manual review before fuzzy backfill

Generate a CSV of fuzzy / town_model proposals: `asset_code, asset_customer, asset_town, source_customer, source_town, match_tier, contract_type, contract_start, contract_end`. Admin reviews and approves the CSV; only approved rows are written. All writes log a history row with `note = 'csv_reviewed_by:<email>'`.

### 9.4 Fallback for unmatched assets

The `contract_type` check constraint does not allow an `'unknown'` value, so assets that don't match any source row are left with **no `asset_lifecycle` entry** — matching today's "No Contract" UI bucket. The history table records `event = 'backfill'`, `note = 'no_match'` for traceability.
