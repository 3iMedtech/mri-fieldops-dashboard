## 2. Exact Schema Proposal

See [`db/migrations/0004_asset_lifecycle_phase1_REVIEW_ONLY.sql`](../db/migrations/0004_asset_lifecycle_phase1_REVIEW_ONLY.sql) for the full DDL. Summary:

### `public.config_assets` — additive columns

| Column | Type | Default | Notes |
|---|---|---|---|
| `status` | `text` | `'active'` | `check (status in ('active','de_installed'))` |
| `de_installed_at` | `timestamptz` | null | Set when status flips to de_installed |
| `de_installed_by` | `uuid` | null | Auth uid of the user who de-installed |
| `created_by` | `uuid` | null | Auth uid; null for legacy XLSX rows |
| `created_at` | `timestamptz` | `now()` | |
| `updated_by` | `uuid` | null | Touched on UPDATE via trigger |
| `updated_at` | `timestamptz` | `now()` | Maintained by `_touch_updated_at()` trigger |
| `note` | `text` | null | Free-text admin note (de-install reason etc.) |

Two filtered indexes added: `status` (covers all status filters) and `where status = 'active'` (covers the dominant active-only query).

### `public.asset_lifecycle` (new)

One **active** contract row per asset_code, plus historical superseded/cancelled rows.

```
id              uuid pk default gen_random_uuid()
asset_code      text not null fk -> config_assets(code)
contract_type   text check in ('warranty','extended_warranty','cmc','labour_contract')
pm_required     boolean        -- true for warranty/extended_warranty/cmc, false for labour_contract
contract_start  date
contract_end    date
status          text check in ('active','superseded','cancelled') default 'active'
source_customer text           -- raw customer string from PM/CMC source for trace
source_ref      text           -- e.g. PM_SCHEDULE.id or CMC_DATA.sn
created_by      uuid
created_at      timestamptz
updated_by      uuid
updated_at      timestamptz
note            text
```

Constraints:
- `asset_lifecycle_pm_required_consistency` — pm_required must follow contract_type rule.
- `asset_lifecycle_dates_sane` — if both dates present, end ≥ start.
- **Partial unique index** `asset_lifecycle_one_active_per_asset on (asset_code) where status = 'active'` — enforces "one active lifecycle per asset" without blocking historical rows.

### `public.asset_lifecycle_history` (new)

Append-only event log.

```
id            uuid pk
asset_code    text not null
lifecycle_id  uuid                -- nullable for asset-only events
event         text in ('created','updated','renewed','de_installed','reactivated','cancelled','backfill')
before_state  jsonb
after_state   jsonb
actor         uuid
actor_email   text
source        text                -- 'app:install_base_form', 'sql:backfill', etc.
occurred_at   timestamptz default now()
note          text
```

Indexes: `(asset_code, occurred_at desc)` for per-asset timeline queries; `(lifecycle_id, occurred_at desc)`; `(event)`.

### Triggers

`_touch_updated_at()` BEFORE UPDATE on both `config_assets` and `asset_lifecycle`.

## 4. Manager Permission — Client-Side Today; DB Will Not Know Without Role Mapping

**Current state (v1.4.0.1):**
- `canManagePM()` in client code returns true for `_userRole === 'admin' || _userRole === 'manager'` **OR** `email === 'manager@3imedtech.com'` (the email-allowlist fallback).
- DB has **no knowledge** of this allowlist. There are no policies, helpers, or tables tracking who is a "manager" at the DB layer today.

**What this means for Phase 1 / Phase 2:**
- The placeholder DB policies in the migration grant write to JWT role ∈ {admin, manager}. The manager-allowlist user does **not** have `role: 'manager'` in their JWT (verified during v1.4.0.1 staging tests: `_userRole === 'viewer'` for `manager@3imedtech.com`).
- Therefore: if Phase 2 ships Add/Edit/De-install UI gated only by `canManagePM()`, the client will let the manager-allowlist user click the button, but PostgREST will reject the insert/update with an RLS error. **This is a UX bug waiting to happen.**

**Blocker before enabling Manager DB writes** (must resolve one of the following):

1. Move the manager allowlist into JWT metadata. Set `role: 'manager'` on the user via Supabase Auth dashboard for `manager@3imedtech.com`. Drop the email-fallback in `canManagePM()`. Pure JWT design (Option A).
2. Adopt Option B (`user_roles` table) and insert a row mapping the allowlist user's `auth.uid()` to `role: 'manager'`.
3. Adopt Option C (email-allowlist in DB) and add the email to a DB-side allowlist policy.

Whatever the choice, **DB-side Manager write policies must be in place AND the client `canManagePM()` must be aligned with them** before any Manager-write button is wired up. Otherwise a Manager will see a button, click it, and get a silent or noisy RLS rejection.

## 5. Asset Code Immutability (Decided Policy #6)

**Decision:** asset_code is immutable after creation.

**DB enforcement:**
- `asset_lifecycle.asset_code` FK uses `on update restrict on delete restrict`. A SQL `update config_assets set code = 'NEW'` would fail if any lifecycle row references the old code. Code rename is forbidden at the DB layer.
- `config_assets` itself: today there is no PK/UNIQUE on `code`; migration §0 adds a unique index defensively. We do NOT add a CHECK constraint or trigger preventing UPDATE of `code` on `config_assets` itself, because cross-table FK enforcement is sufficient and a per-row trigger would slow XLSX bulk upload.

**App enforcement:**
- Add Asset form: `code` field is editable (required) at create time.
- Edit Asset form: `code` field is **read-only** (rendered as plain text or disabled `<input>`).
- The future Edit form's submit handler must NOT include `code` in the upsert payload — even an unchanged value sent through `update` could be safe, but excluding it removes a class of accidental overwrite.

**Test plan:**
- After Phase 2 UI ships, attempt `update public.config_assets set code = 'X' where code = 'Y'` via SQL editor as service_role: must succeed only when no asset_lifecycle row references `Y`. If any reference exists, must fail with FK violation.
- App-side: open Edit Asset for a code with lifecycle history; the code field MUST render read-only. Attempt to bypass via DevTools (set `<input>` value, submit): the submit handler should not send `code` in the payload.

## 6. XLSX Bulk Upload — Must-Resolve Before Add Asset UI Ships

This is now a **must-resolve item**, not a Phase 2 deferral. Without resolution, any app-created asset will be silently destroyed by the next XLSX upload.

### 6.1 Current behavior (the risk)

`index.html:2635` runs:
```js
const delAssets = await _sb.from('config_assets').delete().neq('code', '__sentinel__');
const { error } = await _sb.from('config_assets').insert(deduped);
```

This bulk `delete + insert` flow:
- Wipes EVERY row in `config_assets`, including app-created ones.
- Re-inserts only the rows in the uploaded XLSX.
- Loses `status`, `de_installed_at`, `created_by`, `note`, all v1.4.1 lifecycle fields.
- FK from `asset_lifecycle.asset_code` blocks `delete()` if any lifecycle rows exist (because we set `on delete restrict`). XLSX upload will silently fail or partially fail once Phase 1 DDL is applied and any lifecycle rows have been written.

### 6.2 Required design decision (must resolve before Add Asset UI ships)

Pick one. All three are viable.

| Option | Behavior | Pro | Con |
|---|---|---|---|
| **A — Upsert by code** | Replace `delete + insert` with `upsert(deduped, {onConflict: 'code'})`. Rows in XLSX are updated; rows NOT in XLSX are left alone. App-created rows survive. | Simplest. Preserves all DB-only state. | XLSX no longer "the truth" — admin must explicitly de-install in app or via SQL. Removed XLSX rows are NOT auto-removed from DB. |
| **B — Preserve `app_created` rows during full replace** | Keep current `delete + insert`, but exclude rows where `created_by IS NOT NULL` (or a future `app_created = true` flag) from the delete. | Closest to current XLSX-as-truth model. | Two classes of rows (xlsx-managed vs app-managed) — semantics get complex. FK still blocks delete if app-created assets reference lifecycle rows. |
| **C — Disable XLSX full replace once app CRUD is enabled** | XLSX upload becomes import-only (creates rows only if not present); never deletes. Admin manages mutations through UI exclusively post-Phase 2. | Cleanest separation of concerns. | Largest behavioral change; XLSX flow must be re-validated. |

**Recommendation (for review):** Option A. Smallest code change; survives mixed XLSX + app-created world; FK constraint behaves consistently.

### 6.3 Blocker statement

**The Add Asset UI MUST NOT ship until one of A/B/C is implemented and verified.** Otherwise the first XLSX upload after Add Asset goes live will erase all app-created assets and trigger RLS/FK errors.

## 9. Backfill Strategy

`asset_lifecycle` starts empty. We need historical contract rows from the existing PM_SCHEDULE and CMC_DATA tables to populate it so PM scheduling derived from `pm_required` and renewal urgency derived from `contract_end` keep working.

### 6.1 SQL exact-match phase (server-side, fast, reviewable)

Insert one `asset_lifecycle` row per (`config_assets`, source) pair where:
- exact match: normalized customer key (`lower(regexp_replace(...))`) AND normalized town match
- contract_type derived from PM/CMC `contract` field (`CMC` → 'cmc', `Warranty` → 'warranty', `Extended Warranty` → 'extended_warranty', else flag for review)
- contract_start / contract_end taken from source row
- `event = 'backfill'`, `source = 'sql:backfill_v1'`, `status = 'active'`
- skip if a row already exists for this asset_code

The exact-match SQL will be drafted in Phase 1.5 once schema is approved. It will be run in a transaction with an explicit `select count(*)` before COMMIT.

### 6.2 Node helper for fuzzy + town_model cases

Rows that don't exact-match get fed through the JS layered matcher (`_findContractForAsset` exact → fuzzy → town_model) running in a Node script that connects via the service role and writes the chosen lifecycle row + a history event with `source = 'node:backfill_fuzzy'`.

### 6.3 Manual review before fuzzy backfill

Generate a CSV of fuzzy / town_model proposals: `asset_code, asset_customer, asset_town, source_customer, source_town, match_tier, contract_type, contract_start, contract_end`. Admin reviews and approves the CSV; only approved rows are written. All writes log a history row with `note = 'csv_reviewed_by:<email>'`.

### 6.4 Fallback

Assets with no exact / fuzzy / town_model match get a row with `contract_type = 'unknown'` … **no, scratch that** — the schema doesn't allow `unknown`. Instead, leave them with no `asset_lifecycle` row (matches today's "No Contract" UI bucket). The history table records `event = 'backfill'`, `note = 'no_match'` for traceability.

## 12. Risk Review

|---|---|---|---|
| RLS JWT claim path mismatch (A2) | Medium | High | Verify on staging by inspecting a JWT in the SQL editor: `select auth.jwt()`. Adjust policy expressions before apply if the role lives elsewhere. |

| Backfill misclassifies contract_type | Medium | Medium | Manual CSV review before fuzzy writes. Every backfill row has a history entry with `source = 'sql:backfill_v1'` or `'node:backfill_fuzzy'` so they're identifiable for re-run. |
| Backfill duplicates rows on re-run | Low | Medium | Backfill SQL is idempotent: `where not exists (select 1 from asset_lifecycle where asset_code = ... and status='active')`. |
| Partial unique index conflicts during renewal flow | Low | Medium | Renewal is a 2-step write inside a single transaction: `update old_row set status='superseded'; insert new_row with status='active';`. App must wrap both in a Supabase RPC or use serializable isolation. |
| Cascade delete from config_assets removes asset_lifecycle rows | Low | High | FK is `on delete restrict`. Deletes are denied at the policy layer anyway. |
| Schema migration mis-applied to prod before staging verification | Low | Critical | Approval gate at end of §10. No automation runs DDL. |
| Future `app_user_role()` helper change diverges from inline policies | Low | Low | Document the assumption; consolidate in a v1.4.2 migration if helper is introduced. |
