## 6. Step D — Role-Context Verification (per role)

## 9. Risks (staging-specific)

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| §1.3 V2 audit shows >1 missing rows (unexpected) | Low | Medium | Investigate. Migration 0005 will still INSERT them (idempotent), but operator should understand why staging diverged. |
| §3 Step A applies but §5 Track B deploy is delayed | Low | Low | No bug — Manager's existing UI buttons (limited to PM Save / Reminder via `canManagePM()` allowlist) continue to work. New buttons won't be visible until Track B deploys. |
| Track B deploy lands but §3 Step A failed/skipped | Medium | Medium | Manager clicks Edit Asset → PostgREST 401 / RLS error. UI must show a clear error message. Mitigated by runbook ordering. |
| RPC `app_user_role` fails on Track B boot | Low | Medium | App falls back to JWT path; user sees v1.4.0.1 behavior for that session. Hard refresh retries. |
| 0005 backfill row data divergence | Low | Medium | Row data taken verbatim from `index.html:1472-1498`. If V2 ever changes in source, regenerate the migration. |
| Manager XLSX upload now possible — accidentally enabled | Medium | Low | Per §2.5 of phase2_review.md, this is the recommended Option (a). If reviewer disagrees, gate the upload button by `_userRole === 'admin'` in the Track B patch. |
