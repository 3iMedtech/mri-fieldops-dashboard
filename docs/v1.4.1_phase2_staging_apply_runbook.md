-- G: GRANT / table-level privilege check for the `authenticated` role.
--
-- Migration 0004 adds RLS policies, but RLS only takes effect if the
-- table-level GRANT for the role allows the operation in the first
-- place. If `authenticated` lacks INSERT or UPDATE on a table, the
-- new RLS policies cannot be exercised — PostgREST returns a 401
-- BEFORE RLS is evaluated.
--
-- This check confirms `authenticated` already has the required GRANTs
-- on the 3 legacy tables we're about to add app_can_write() policies
-- to. Phase 1 inspection (2026-05-09) showed legacy admin_* policies
-- in place; those policies imply the GRANTs were set at original
-- table creation. We re-verify here in case anything has drifted.
select t                                                       as table_name,
       has_table_privilege('authenticated', t::regclass, 'INSERT') as can_insert,
       has_table_privilege('authenticated', t::regclass, 'UPDATE') as can_update,
       has_table_privilege('authenticated', t::regclass, 'DELETE') as can_delete
  from unnest(array['public.config_assets','public.pm_schedule','public.cmc_contracts']) as t
 order by t;
-- expected: can_insert=true AND can_update=true for all 3 tables.
-- can_delete may be true or false; not required for Phase 2 (no DELETE
-- policy is added — see migration 0004 §1/§2/§3 comments).
--
-- If can_insert=false or can_update=false on any row: STOP. The new
-- RLS policies will not actually permit Manager writes because the
-- table-level GRANT denies before RLS even runs. Adding explicit
-- `grant insert, update on public.<table> to authenticated;` would
-- be required, and that change should be reviewed separately before
-- proceeding with 0004.
```

-- Cleanup the synthetic row after the test.
--
-- ⚠️ IMPORTANT: this DELETE MUST be run from the default `postgres` /
-- service-role SQL Editor session — NOT from an authenticated app
-- session (Admin / Manager / Engineer signed-in via PostgREST).
--
-- Why: Phase 2 migration 0004 INTENTIONALLY does not add a DELETE
-- policy via `app_can_write()`. Authenticated app sessions cannot
-- DELETE config_assets via PostgREST regardless of role:
--   - Admin satisfies legacy `admin_delete_config_assets` (JWT
--     `app_metadata.role='admin'`), so an Admin's PostgREST DELETE
--     would in principle work — but the legacy policy is the only
--     gate, and we deliberately route operational DELETEs through
--     SQL Editor for traceability.
--   - Manager and Engineer have no DELETE policy at all and would
--     hit `permission denied for table config_assets` (table-level
--     GRANT) or RLS denial.
-- Phase 2's de-installation pattern is UPDATE `status='de_installed'`,
-- NOT DELETE. The TEST-PHASE2-RGR row is a regression-test artifact;
-- removing it via service-role here is the correct cleanup path.
delete from public.config_assets where code = 'TEST-PHASE2-RGR';
```

(If `asset_lifecycle` has rows referencing it, the FK `on delete restrict` blocks the delete. In that case either run a service-role DELETE on the referencing lifecycle rows first OR leave the synthetic asset in place and tag it for de-install via the lifecycle workflow instead.)
