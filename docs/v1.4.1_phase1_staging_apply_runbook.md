### 3.3 Last-admin guard tests — SESSION CONTEXT IS LOAD-BEARING

⚠️ **Pre-requisite:** the `v141_user_roles_write_admin` policy must reference the security-definer helper `public._other_active_admin_exists(uuid)`. The 2026-05-09 staging §C.1 run uncovered an RLS recursion bug in the original inline-EXISTS draft of the WITH CHECK clause. The fix is in 0003 §1b/§1c of the migration. If you applied 0003 BEFORE the fix landed, run the staging hot patch (provided alongside this runbook) before §3.3.

⚠️⚠️⚠️ **READ THIS BEFORE PASTING ANY SQL IN §3.3.**

These tests verify that RLS blocks the sole admin from being demoted, disabled, or deleted. **The Supabase SQL Editor by default runs as `postgres` (service-role-equivalent), and `postgres` BYPASSES RLS.** If you run §3.3a/b/c in default SQL Editor without switching session context, the UPDATE/DELETE statements will **succeed**, demoting/deleting the production admin and locking everyone out. Recovery requires a service-role direct SQL fix.

**To run these tests safely, use ONE of these three approaches:**

**Approach 1 — Authenticate as Admin in the SQL Editor (preferred):**
- Use Supabase Studio's "Authenticate" dropdown above the SQL Editor and pick the admin user (`abhijit.s@3imedtech.com`). All queries in that pane now run with the admin's JWT and respect RLS.
- (If your Supabase plan doesn't expose this dropdown, use Approach 2 or 3.)

**Approach 2 — Wrap in `BEGIN ... ROLLBACK` (works in any session):**
- Wrap the entire §3.3 block in a transaction that always rolls back. Even if RLS is bypassed (because you're running as `postgres`), the demote/delete is undone before commit.
- This sacrifices the "test that RLS rejected" signal — you'll see 1 row updated regardless. But it's safe.

**Approach 3 — Run from the staging app's browser console as Admin:**
- Sign in as Admin to the staging app. Open browser DevTools console. Run e.g. `await _sb.from('user_roles').update({role:'viewer'}).eq('user_id', '<admin-uuid>')` and inspect the response (which will be empty / RLS-rejected).

**3.3a/b/c — sole-admin self-demote/disable/delete must be REJECTED**

If using Approach 1 or 3, run as-is. If using Approach 2, wrap as shown:

```sql
-- ⚠️ APPROACH-2 SAFETY WRAPPER — required if running in default postgres session.
-- Skip these BEGIN/ROLLBACK lines if you've authenticated as Admin via Approach 1.
BEGIN;

-- 3.3a: sole admin tries to self-demote — MUST FAIL under RLS
update public.user_roles set role = 'viewer'
 where user_id = (select id from auth.users where email='abhijit.s@3imedtech.com');
-- expected as Admin (Approach 1/3): 0 rows updated (RLS WITH CHECK rejects)
-- expected as postgres (Approach 2): 1 row updated (RLS bypassed; will be rolled back below)
-- IMPORTANT: if Approach 1/3 returns "infinite recursion detected in policy
--   for relation user_roles", the recursion-fix patch has NOT been applied.
--   Apply the staging hot patch (creates _other_active_admin_exists helper +
--   recreates the policy) before continuing.

-- 3.3b: sole admin tries to self-disable — MUST FAIL under RLS

 where user_id = (select id from auth.users where email='abhijit.s@3imedtech.com');
-- expected as Admin: 0 rows updated
-- expected as postgres: 1 row updated (rolled back)

 where user_id = (select id from auth.users where email='abhijit.s@3imedtech.com');

ROLLBACK;
-- safety net: any successful 3.3a/3.3b row mutation is undone here
```

**3.3d — promote a real existing user to a temporary second admin (REPLACES old gen_random_uuid approach)**

⚠️ The earlier draft of this runbook used `gen_random_uuid()` for the temp admin's user_id, but `user_roles.user_id` has a FK to `auth.users(id)` and FK constraints fire even for service_role. **Use this corrected approach instead:**

**3.3e — repeat the demote/disable on the original admin; now they should SUCCEED**

```sql
-- Wrap in BEGIN/ROLLBACK because we don't want to leave the admin demoted.
BEGIN;

-- 3.3e-i: demote original admin while a second admin exists — should SUCCEED
update public.user_roles set role='viewer'
 where user_id = (select id from auth.users where email='abhijit.s@3imedtech.com');
-- expected: 1 row updated (RLS allows because engineer is now also admin)

-- (We rely on the 3.3d temp promotion still being in place after rollback;

```

## 9. What Happens at Each Stop Point

When operator approves the next step, the assistant will:

1. **§1 pre-flight:** post the SQL queries; operator runs them; operator pastes the output. Assistant parses and confirms against expected values.
2. **§2.1 Step A:** post the migration content (or instructions to copy/paste from disk); operator runs it; operator pastes the success/error output.
3. **§2.2 Step B:** post verification queries; operator runs + pastes; assistant parses.
4. **§2.3 Step C:** post the seed SQL; operator runs as service_role; operator pastes output.
5. **§3 post-apply:** post each verification query; operator runs + pastes; assistant parses and tells operator whether all green.
6. **§4 app role verification:** post test plan; operator signs in to each role and runs; operator pastes the role-helper RPC output and any visible-app behavior. Assistant cross-checks against §5 expected results.
7. **§6 rollback (only if needed):** post pre-rollback snapshot SQL → operator saves → assistant posts rollback file content → operator runs.

The assistant never runs SQL itself in this runbook. **Operator always controls the execute button.**
