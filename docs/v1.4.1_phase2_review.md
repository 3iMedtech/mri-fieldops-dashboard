## 1. Phase 2 Scope

The Phase 2 review package ships in **two coordinated tracks** that must deploy together.

### Track A — Database (one new migration: `0004_*`)

- **Additive write policies** on the three legacy tables. Manager (and Admin) writes via the new `app_can_write()` policies; the legacy `admin_*` policies remain in place. PostgreSQL OR's permissive policies, so any user matching either gate can write.
- **Install Base master-source backfill** (separate migration `0005_*`) — one-time INSERT of any V2 rows missing from `config_assets` so DB matches `INSTALL_BASE_V2.length` (currently 25 codes vs 24 in production / staging).

### Track B — App runtime (single coordinated `index.html` patch)

- **App reads role via `_sb.rpc('app_user_role')`** instead of `app_metadata.role`. Caches in `_userRole`. Refreshes on auth state change. Falls back to 'viewer' if RPC fails.
- **`canManagePM()` simplified** — drops the email allowlist branch. Reduces to `_userRole === 'admin' || _userRole === 'manager'`.
- **XLSX upload rewrite** at `index.html:2635` — `delete + insert` → `upsert(rows, { onConflict: 'code' })`. Excludes lifecycle fields from the upsert payload.
- **Lifecycle UI workflows** — Add Asset, Edit Asset, De-install Asset, Renew Contract. Buttons gated by `canManagePM()`. Writes via the new policies in Track A.

### Sequencing constraint

**Track A SQL must be applied BEFORE the app deploys with Track B code.** Otherwise:
- Manager clicks "Edit Asset" → app sends UPDATE → legacy `admin_update_config_assets` rejects → user sees an error.

After Track A SQL applies, Manager's UPDATE goes through the new additive policy. After Track B app deploys, `_userRole='manager'` (from the RPC) instead of `'viewer'` (from the JWT), and the email allowlist is no longer needed.

The runbook ([`v1.4.1_phase2_staging_apply_runbook.md`](v1.4.1_phase2_staging_apply_runbook.md)) sequences this explicitly.

> **Renew workflow has its own sequencing dependency.** The Renew RPC `public.renew_asset_lifecycle(...)` referenced in §7.4 is **NOT** part of this Phase 2 SQL package (PR #26 ships only `0004_*` additive write policies + `0005_*` IB master backfill). Creating that RPC is a follow-on migration (proposed `0006_v141_phase2_renew_lifecycle_rpc_*.sql`) that must land in its own review/apply gate **before** the Track B Renew button can be wired up in `index.html`. Add Asset / Edit Asset / De-install do NOT depend on this RPC and can ship as part of Track B even if the Renew RPC is deferred. See §7.4 for the RPC contract and §12 Q2 for the open scope question.

## 2. Track A.1 — Additive Write Policies (Migration `0004_*`)

### 2.1 What it does

Adds 6 new RLS policies (3 tables × 2 commands) on `public.config_assets`, `public.pm_schedule`, and `public.cmc_contracts`:

| Table | Policy | Command | Gate |
|---|---|---|---|
| config_assets | `v141_config_assets_insert_app_can_write` | INSERT | `with check (public.app_can_write())` |
| config_assets | `v141_config_assets_update_app_can_write` | UPDATE | `using (public.app_can_write()) with check (public.app_can_write())` |
| pm_schedule | `v141_pm_schedule_insert_app_can_write` | INSERT | same |
| pm_schedule | `v141_pm_schedule_update_app_can_write` | UPDATE | same |
| cmc_contracts | `v141_cmc_contracts_insert_app_can_write` | INSERT | same |
| cmc_contracts | `v141_cmc_contracts_update_app_can_write` | UPDATE | same |

**DELETE is NOT added** as an `app_can_write()` policy. Existing `admin_delete_*` policies (JWT app_metadata.role='admin') retain admin-only DELETE. De-installation is via UPDATE of `config_assets.status='de_installed'`, not DELETE.

### 2.2 Why additive, not migrate

Phase 1 review §3.7 documented the choice:
- **Option B-Migrate** would drop the legacy `admin_*` policies and create only the `v141_*_app_can_write` versions. Single source of truth, cleanest. But during the migration window, an admin's writes could fail if the legacy policy is dropped before the new one lands.
- **Option B-Additive** keeps both. PostgreSQL OR's permissive policies, so admin satisfies either gate. Manager satisfies only the new `app_can_write()` gate. Lowest-risk.

Recommendation from Phase 1 review §3.7: start with Additive. After 1-2 weeks of clean operation, run a follow-up migration (`0006_*` or later) to drop the legacy `admin_*` write policies as a final consolidation. **Phase 2 ships the Additive step only.**

### 2.3 Files

- [`db/migrations/0004_v141_phase2_additive_write_policies_REVIEW_ONLY.sql`](../db/migrations/0004_v141_phase2_additive_write_policies_REVIEW_ONLY.sql)
- [`db/migrations/0004_v141_phase2_additive_write_policies_ROLLBACK_REVIEW_ONLY.sql`](../db/migrations/0004_v141_phase2_additive_write_policies_ROLLBACK_REVIEW_ONLY.sql)

### 2.4 Idempotency + Safety

- All 6 policies are guarded with `drop policy if exists` + `create policy`.
- Rollback drops only the 6 new policies; legacy `admin_*` policies stay untouched.
- No DDL on data — pure policy add. Existing rows / writes by Admin (JWT) continue to work.

### 2.5 Manager XLSX upload — separate decision

After this migration applies, Manager has DB-write rights to `config_assets` via `app_can_write()`. So if Phase 2 also lets Manager click the XLSX upload button, the upload would land. Phase 1 review §6.3 explicitly deferred this:

> **Do NOT enable Manager XLSX permission in Phase 1.** Phase 2 must explicitly decide whether Manager should be able to upload XLSX too.

**Decision needed before Phase 2 ships:**
- (a) Allow Manager XLSX upload — gate the upload button by `canManagePM()` (default behavior once the email allowlist is removed). Phase 2 ships this naturally if no UI gating is added.
- (b) Restrict XLSX upload to Admin only — gate the upload button by `_userRole === 'admin'`, even after the legacy `admin_write_config_assets` policy is bypassable by Manager.

**Recommendation:** Option (a) — Manager has parity per product decision #8 (Manager has parity with Admin/Superadmin for Add/Edit/De-install/Lifecycle). XLSX upload is a bulk version of the same write surface; restricting it to Admin only would be inconsistent. Document explicitly so reviewers can disagree.

## 3. Track A.2 — Install Base Master-Source Backfill (Migration `0005_*`)

### 3.1 What it does

One-time INSERT of any rows from the hardcoded `INSTALL_BASE_V2` array (in `index.html:1472-1498`) that are missing from `public.config_assets`. After this migration applies, `select count(*) from config_assets` should equal `INSTALL_BASE_V2.length` (25).

### 3.2 Why

Per Phase 1 review §1 (Source-of-truth callout) and the resolved DB-is-master decision:

- Production / staging `config_assets` has 24 rows.
- `INSTALL_BASE_V2` in source has 25 rows (codes `AN001`..`AN025`).
- Pre-Phase 2 overlay merges V2 + DB → app shows 25 rows even though DB has 24.
- **Decision:** DB is master. The missing V2 row(s) must be backfilled into `config_assets` so the DB count matches V2.length, and V2 becomes a frozen cold-start fallback only.

### 3.3 How

The migration uses one INSERT statement with a 25-row `(values ...)` clause, gated by `where not exists (select 1 from public.config_assets where code = ?)`. So:
- Codes already in DB are skipped silently.
- Codes missing from DB are inserted with full provenance: `created_by = null`, `note = 'v1.4.1 phase 2 install_base_v2 backfill'`, `status = 'active'`, `created_at`/`updated_at` defaulted.

This is idempotent — re-running the migration is safe.

### 3.4 Files

- [`db/migrations/0005_v141_phase2_install_base_master_backfill_REVIEW_ONLY.sql`](../db/migrations/0005_v141_phase2_install_base_master_backfill_REVIEW_ONLY.sql)
- [`db/migrations/0005_v141_phase2_install_base_master_backfill_ROLLBACK_REVIEW_ONLY.sql`](../db/migrations/0005_v141_phase2_install_base_master_backfill_ROLLBACK_REVIEW_ONLY.sql)

### 3.5 Rollback

Rollback removes only rows tagged with `note = 'v1.4.1 phase 2 install_base_v2 backfill'`. Pre-existing rows are not touched. If the operator manually edits the inserted rows post-backfill to remove the marker note, those rows become invisible to the rollback — by design (the operator chose to keep them).

### 3.6 Sequencing relative to 0004

Migration 0005 has no policy dependency. It can run before, after, or between 0004 statements. The runbook applies them in order 0004 → 0005 to match the migration numbering.

### 3.7 Open data-quality task: identify the missing code(s) before commit

Before approving migration 0005, the operator should run on staging AND production:


## 4. Track B.1 — App RPC Integration

### 4.1 Current state ([`index.html:8097`](../index.html:8097))

```js
function applyRoleRestrictions(){
  const metaRole = _currentUser?.app_metadata?.role;
  _userRole = metaRole === 'admin' ? 'admin'
            : metaRole === 'manager' ? 'manager'
            : 'viewer';
  // ...rest of the function...
}
```

`_userRole` is a synchronous read from the JWT's `app_metadata.role`. Manager-allowlist user has `app_meta_role='viewer'`, so `_userRole='viewer'` for them — the Phase 1.5 gap.

### 4.2 Proposed patch (review-only)

Make `applyRoleRestrictions` async and call the RPC. Cache in `_userRole`. Fall back to JWT path if RPC fails.

```js
async function applyRoleRestrictions(){
  // Phase 2: read role from public.user_roles via SECURITY DEFINER RPC.
  // Falls back to JWT app_metadata.role if RPC errors (e.g. transient
  // network blip during boot).
  let resolvedRole = 'viewer';  // safe default
  try {
    const { data, error } = await _sb.rpc('app_user_role');
    if(!error && (data === 'admin' || data === 'manager' || data === 'viewer')){
      resolvedRole = data;
    } else if(error){
      console.warn('app_user_role RPC failed; falling back to JWT', error);
      const metaRole = _currentUser?.app_metadata?.role;
      resolvedRole = metaRole === 'admin' ? 'admin'
                   : metaRole === 'manager' ? 'manager'
                   : 'viewer';
    }
  } catch(_){
    const metaRole = _currentUser?.app_metadata?.role;
    resolvedRole = metaRole === 'admin' ? 'admin'
                 : metaRole === 'manager' ? 'manager'
                 : 'viewer';
  }
  _userRole = resolvedRole;
  // ...rest of the function continues unchanged...
}
```

### 4.3 Call sites

`applyRoleRestrictions()` is called after Supabase auth resolves a session. Need to verify all call sites are await-able. Most call sites are inside async handlers already. Any synchronous call site needs to be wrapped in `void applyRoleRestrictions()` or migrated to async.

To be confirmed during implementation review (not Phase 2 review). Plan: grep for `applyRoleRestrictions(` and inspect each call site's context.

### 4.4 Caching + refresh

- Cache: `_userRole` (existing global). Refresh on auth state change (Supabase fires `auth.onAuthStateChange` with `'SIGNED_IN'`/`'SIGNED_OUT'` — re-run `applyRoleRestrictions()` on each).
- Stale-on-promote: if Admin promotes Engineer to Manager via the future Admin UI, Engineer's existing session continues to show `_userRole='viewer'` until they refresh the page or sign out and back in. Acceptable for v1.4.1; Phase 3 could add a real-time channel subscription on `user_roles` if needed.

### 4.5 Risk

- **RPC failure on boot** = user sees `_userRole='viewer'` (fallback default) and Admin UI elements are hidden. Hard refresh or sign-in cycle resolves. Mitigated by JWT-fallback inside the function.
- **RPC latency** adds ~50-200ms to login flow. Acceptable.

### 4.6 Test plan

- Sign in as Admin → `_userRole='admin'`, RPC ran successfully (verify via console).
- Sign in as Manager → `_userRole='manager'` (NOT 'viewer'). The Phase 1.5 gap is now closed.
- Sign in as Engineer → `_userRole='viewer'`.
- Force RPC failure (block `*/rpc/app_user_role` in DevTools network panel) → verify fallback to JWT path; `_userRole` resolves correctly for Admin (whose JWT has `app_metadata.role='admin'`); Manager would resolve to `'viewer'` under fallback (matching v1.4.0.1 behavior); Engineer to `'viewer'`.

## 6. Track B.3 — XLSX Upload Rewrite

### 6.1 Current state ([`index.html:2635-2637`](../index.html:2635))

```js
const delAssets = await _sb.from('config_assets').delete().neq('code', '__sentinel__');
const { error } = await _sb.from('config_assets').insert(deduped);
```

Bulk `delete + insert`. Wipes all rows, re-inserts XLSX rows. Loses status, de_installed_at, created_by, note, etc. Once `asset_lifecycle` has rows referencing `config_assets`, the FK `on delete restrict` blocks the delete entirely — the XLSX upload would fail.

### 6.2 Proposed patch (review-only)

Per Phase 1 review §6.3 (Option A — Upsert by code, decided 2026-05-09):

```js
// Phase 2: upsert-by-code instead of delete+insert. Preserves app-created
// rows and lifecycle fields. The XLSX-owned columns are explicit so the
// upsert never overwrites status/de_installed_at/created_by/note/etc.
const xlsxOwnedColumns = ['code','ast_id','name','tesla','model','town','state',
                          'channel','gradient','sw','compressor','coldhead','alias_of'];
const upsertRows = deduped.map(r => {
  const out = {};
  for(const k of xlsxOwnedColumns) out[k] = r[k] ?? null;
  return out;
});
const { error } = await _sb
  .from('config_assets')
  .upsert(upsertRows, { onConflict: 'code' });
if(error){ /* existing error handling */ }
// NOTE: rows in config_assets but NOT in deduped are LEFT IN PLACE.
// De-installation must happen via the lifecycle workflow, not by XLSX omission.
```

### 6.3 Pre-flight de-install audit (UI improvement)

Before each XLSX upload, the app should generate a diff:

- **In XLSX, not in DB** → will be INSERTed.
- **In both** → will be UPDATEd (XLSX-owned columns only).
- **In DB, not in XLSX** → SKIPPED (NOT auto-deleted). Surfaced as a "candidate de-install list" for Admin to review.

Implementation: a confirmation modal showing the three lists with row counts before the upsert runs.

### 6.4 Test plan

- Insert a synthetic asset in `config_assets` directly (`TEST-IB-AAA`) with `status='active'`, `note='manual app insert'`. Run an XLSX upload that does NOT include `TEST-IB-AAA`. Verify post-upload that `TEST-IB-AAA` still exists with `note='manual app insert'`. **Critical regression test.**
- Run an XLSX upload that includes a row matching an existing `config_assets.code` but with a different `name`. Verify `name` is updated; `status`, `de_installed_at`, `created_by`, `note` are NOT changed.
- Run an XLSX upload that includes a brand-new `code` not in DB. Verify it's INSERTed.
- Manager runs XLSX upload (after migration 0004 landing). Verify success at PostgREST. Verify the audit list of "in DB, not in XLSX" is correctly surfaced.

### 6.5 Risk

- **Existing app behavior change.** Today's XLSX is "the truth"; after Phase 2 it's "additive truth". Customer expects same behavior — needs a brief reviewer sign-off / docs note.
- **FK constraint** — already mitigated by switching from delete to upsert; FK no longer triggered.

## 7. Track B.4 — Lifecycle UI Workflows

### 7.1 Add Asset

**Where:** Install Base page header. Button visible only to `canManagePM()`.

**Form fields:**
- `code` (required, immutable post-create — text input with format hint)
- `ast_id` (optional)
- `name` (required)
- `tesla`, `model`, `town`, `state` (all optional but recommended)
- `channel`, `gradient`, `sw`, `compressor`, `coldhead` (optional)
- `note` (optional, free text)

**On submit:**
- `_sb.from('config_assets').insert({code, ast_id, name, ..., status: 'active', created_by: auth.uid(), note})`
- Insert a corresponding `asset_lifecycle_history` row: `event='created'`, `source='app:install_base_form'`, `actor=auth.uid()`, `actor_email`, `before_state=null`, `after_state` = the inserted row.
- On success, refresh `CONFIG_ASSETS` (re-run Supabase load).

**Validation:**
- `code` uniqueness — server-side via PK constraint on `config_assets.code`. Client should also pre-check via SELECT to avoid round-trips.
- `code` format — recommend uppercase alphanumeric, 4-12 chars (consistent with V2 `AN001`-`AN025` pattern). Soft validation; not enforced at DB.

### 7.2 Edit Asset

**Where:** Install Base table row → "Edit" action. Visible only to `canManagePM()`.

**Form:**
- All fields editable EXCEPT `code` (rendered as plain text / disabled `<input>`).
- Submit handler explicitly omits `code` from the upsert payload.

**On submit:**
- `_sb.from('config_assets').update({ast_id, name, ..., updated_by: auth.uid(), note}).eq('code', editingCode)`
- Insert `asset_lifecycle_history` row: `event='updated'`, `source='app:install_base_form'`, `before_state` = the row before edit (capture pre-submit), `after_state` = the row after edit.

### 7.3 De-install Asset

**Where:** Install Base row → "De-install" action. Visible only to `canManagePM()`.

**Confirmation:**
- Modal asks operator to type the asset_code to confirm (per product decision #7).
- On confirmation, `_sb.from('config_assets').update({status: 'de_installed', de_installed_at: now(), de_installed_by: auth.uid(), note: '<reason from form>'}).eq('code', code)`.
- Insert `asset_lifecycle_history` row: `event='de_installed'`, `source='app:de_install_form'`, `before_state`/`after_state`.
- Active lifecycle row(s) for this asset are NOT auto-cancelled. Admin can choose to flip them to `cancelled` separately.

### 7.4 Renew Contract

**Where:** Contracts page row → "Renew" action. Visible only to `canManagePM()`.

**Workflow** (2-step transaction in a Supabase RPC OR a 2-statement client side with serializable isolation — Phase 2 ships RPC):

1. UPDATE the existing active `asset_lifecycle` row → `status='superseded'`, `updated_by`, `updated_at`.
2. INSERT a new `asset_lifecycle` row → `status='active'`, with new `contract_type`, `pm_required`, `contract_start`, `contract_end`, `source_ref='app:renew'`, `created_by`.
3. Insert two `asset_lifecycle_history` rows (one per UPDATE / INSERT, with `event='renewed'`).

**RPC name proposal:** `public.renew_asset_lifecycle(asset_code, contract_type, pm_required, contract_start, contract_end, note)`. Returns the new lifecycle id. Wraps both DML calls in a single transaction. Defers to Phase 2 implementation review.

> **⚠️ This RPC is NOT in the Phase 2 SQL package (PR #26).** PR #26 contains only `0004_*` (additive write policies) and `0005_*` (IB master backfill). Creating `public.renew_asset_lifecycle(...)` requires a follow-on migration (proposed file: `db/migrations/0006_v141_phase2_renew_lifecycle_rpc_REVIEW_ONLY.sql` + matching `_ROLLBACK_REVIEW_ONLY.sql`) with its own review-only PR, runbook entry, and approval gate. The Track B Renew UI button in `index.html` MUST NOT be wired up until this follow-on migration has applied to the target environment. Add Asset / Edit Asset / De-install / History viewer (§7.1, §7.2, §7.3, §7.5) have no such dependency and can ship in the first Track B implementation PR; the Renew button can ship in a second Track B implementation PR after `0006_*` lands. See §1 Sequencing constraint.

### 7.5 Lifecycle history viewer

**Where:** Install Base row → "History" action (read-only, available to all roles per `v141_history_select_authenticated`).

Displays the asset_lifecycle_history rows for that asset_code, ordered by occurred_at desc.

### 7.6 Test plan

Per role, verify:
- **Admin:** all 5 actions visible. All 5 succeed via PostgREST + RLS.
- **Manager:** all 5 actions visible (same buttons). Add Asset / Edit / De-install / Renew succeed. History viewer succeeds (read-only).
- **Engineer:** all 5 action buttons hidden. History viewer button — TBD: should engineer see history? Phase 1 RLS allows SELECT for any authenticated user. Recommendation: yes, history viewer visible to engineer (read-only, useful for their own field reference). If Phase 2 disagrees, gate it client-side.

### 7.7 Risk

- **Confirmation typing for De-install** — adds friction but matches product decision #7.
- **Renew RPC** has subtle transactional semantics. Defer detailed review to a follow-on Phase 2 implementation review.
- **Concurrency** — two managers renewing the same asset at roughly the same time. The partial unique index `asset_lifecycle_one_active_per_asset` guarantees only one wins; the other gets a UNIQUE violation error. Acceptable; surface a "concurrent edit" message and re-fetch.
