## 0. Scope Boundary — Reload vs XLSX Replace, and the DB-is-Master Transition

**This patch fixes page-reload preservation only.** It does **not** fix the destructive XLSX bulk upload at [index.html:2635](../index.html:2635) which `delete + insert`s the entire `config_assets` table. As of 2026-05-09 product decisions, two related work items are now scoped explicitly:

| Concern | This patch | Separate work item |
|---|---|---|
| App-created asset disappears after **page reload** | ✅ FIXED | – |
| Lifecycle fields (`status`, `de_installed_at`, `note`, etc.) erased on **page reload** | ✅ FIXED | – |
| App-created asset wiped by **XLSX upload** | ❌ NOT this patch | See [`v1.4.1_phase1_review.md` §6](v1.4.1_phase1_review.md) — **Decision: Option A (upsert-by-code), Phase 2.** Until that lands, Add Asset UI cannot ship. |
| Lifecycle fields wiped by **XLSX upload** | ❌ NOT this patch | Same as above |
| FK from `asset_lifecycle.asset_code` blocks XLSX `delete()` once any lifecycle row exists | ❌ NOT this patch | Same as above |
| 24 (DB) vs 25 (V2) source-of-truth gap | Mitigates by merge-union, does NOT fix the data | Phase 2 one-time backfill of missing V2 row(s) into `config_assets`. See [`v1.4.1_phase1_review.md` §1](v1.4.1_phase1_review.md). |

The reload preservation problem and the XLSX bulk-replace problem share a symptom (Supabase row vanishing) but have separate root causes. **This overlay patch is necessary but NOT sufficient before Add Asset UI ships.** XLSX upsert (§6 Option A) and IB backfill must also land before Phase 2 enables Add Asset UI.

### DB-is-Master Transition Support

Per the 2026-05-09 product decision, **Supabase `public.config_assets` is the future source of truth for Install Base**, and `INSTALL_BASE_V2` becomes a frozen fallback / cold-start seed only. This overlay patch supports that transition in three ways:

1. **Carries forward DB-side lifecycle fields across reload.** Whichever rows exist in DB on reload bring their `status`, `de_installed_at`, `de_installed_by`, `created_*`, `updated_*`, and `note` values forward into `CONFIG_ASSETS`. This is the load-bearing job once DB is master.
2. **Does NOT preferentially overwrite DB physical attrs from V2 once backfill completes.** The current patch logic still has V2 win for physical attrs when both exist (intentional during the transition), but post-backfill every code in V2 also exists in DB, so the merge becomes a no-op for physical attrs and a carry-forward for lifecycle. After Phase 2 / 1.5+, the V2-physical-attr branch is dead code that can be retired in v1.5.
3. **Preserves DB-only codes.** After Phase 2 ships Add Asset UI, the new asset codes will live in DB and not in V2. The merge-union semantics ensure those codes survive every reload.

**INSTALL_BASE_V2 lifecycle after this patch:**
- Stays in source for cold-start render (before Supabase responds).
- Stays as a legacy seed for fresh environments (running migrations on a brand-new project).
- Treated as **frozen** — additions/edits to V2 require a paired DB INSERT.
- v1.5 review can consider extracting V2 to a JSON asset and removing it from `index.html` entirely once the transition is fully validated.

### What this patch does NOT do (binding clarifications)

- **Does NOT fix the XLSX `delete + insert` flow.** That is Phase 2 work, separately tracked in phase1_review.md §6 Option A.
- **Does NOT backfill missing V2 rows into `config_assets`.** That is a Phase 2 SQL one-shot, separately tracked in phase1_review.md §1.
- **Does NOT change `INSTALL_BASE_V2` array contents.** The 25-entry list stays exactly as today; no re-ordering, no removals.
- **Does NOT alter Supabase load query** (`config_assets.select('*')`).
- **Does NOT depend on the `user_roles` migration landing first.** The overlay patch is pure JS; it works pre- or post-DDL. The two are sequenced for safety, not for correctness.

## 1. Why This Patch Exists

`applyInstallBaseOverlay()` is called twice in the app's load lifecycle:

1. **Cold seed** at [index.html:1520](../index.html:1520) — runs immediately so the dashboard can render the 25 hardcoded `INSTALL_BASE_V2` rows before Supabase responds.
2. **Post-Supabase** at [index.html:8202](../index.html:8202), inside the data-load flow, **after** `CONFIG_ASSETS = assets.map(...)` has already populated from `config_assets` table.

The current implementation is destructive on the second call: it iterates `INSTALL_BASE_V2` and overwrites `CONFIG_ASSETS` from scratch. The previous Supabase rows are kept only if their `code` is also in `INSTALL_BASE_V2`, and even then only `open_calls` / `total_calls` are carried forward. Any other Supabase-side state — including the v1.4.1 lifecycle fields (`status`, `de_installed_at`, etc.) and any app-created asset whose `code` is **not** in the hardcoded list — is silently dropped.

Net effect today:
- App cannot reliably persist a new asset to `config_assets` because reload erases it.
- Even if the schema allowed `status = 'de_installed'`, reload would re-create the asset as `'active'`.
- Any v1.4.1 lifecycle UI work depends on this fix landing first.

## 2. Current Code (lines 1500–1517)

```js
// Rebuild CONFIG_ASSETS from INSTALL_BASE_V2 and carry over the {open_calls,
// total_calls} counters from the previous state. Call after any CONFIG_ASSETS
// reassignment (Supabase load, XLSX upload, cold seed).
function applyInstallBaseOverlay(){
  const prev = new Map((CONFIG_ASSETS || []).map(a => [a.code, a]));
  CONFIG_ASSETS = INSTALL_BASE_V2.map(src => {
    const p = prev.get(src.code) || {};
    return {
      code: src.code, ast_id: src.ast_id, name: src.name,
      tesla: src.tesla, model: src.model, town: src.town, state: src.state,
      channel: src.channel, gradient: src.gradient, sw: src.sw,
      compressor: src.compressor, coldhead: src.coldhead,
      _alias_of: null,
      open_calls:  p.open_calls  || 0,
      total_calls: p.total_calls || 0,
    };
  });
}
```

## 3. Proposed Patch

```js
// Merge INSTALL_BASE_V2 (canonical hardcoded baseline) with the current
// CONFIG_ASSETS state (typically just-loaded from Supabase). v1.4.1
// fix: this function used to rebuild from INSTALL_BASE_V2 only, which
// dropped app-created assets and erased lifecycle fields on reload.
//
// Merge rules:
// - Codes present in INSTALL_BASE_V2: use V2 as the canonical source for
//   physical attributes (name/model/town/state/channel/gradient/sw/
//   compressor/coldhead/tesla/ast_id), but CARRY FORWARD the
//   lifecycle/audit fields from prev (status, de_installed_at,
//   de_installed_by, created_*, updated_*, note) and the in-memory
//   counters (open_calls, total_calls).
// - Codes only in prev (= app-created or XLSX-imported assets): keep
//   the prev row as-is. _alias_of is preserved for these rows.
function applyInstallBaseOverlay(){
  const prev = new Map((CONFIG_ASSETS || []).map(a => [a.code, a]));
  const v2  = new Map(INSTALL_BASE_V2.map(s => [s.code, s]));
  const allCodes = new Set([...v2.keys(), ...prev.keys()]);

  CONFIG_ASSETS = Array.from(allCodes).map(code => {
    const src = v2.get(code);    // hardcoded canonical, may be undefined
    const p   = prev.get(code) || {};

    if(src){
      // Code is in INSTALL_BASE_V2 — merge V2 attrs with prev lifecycle.
      return {
        code: src.code, ast_id: src.ast_id, name: src.name,
        tesla: src.tesla, model: src.model, town: src.town, state: src.state,
        channel: src.channel, gradient: src.gradient, sw: src.sw,
        compressor: src.compressor, coldhead: src.coldhead,
        _alias_of: null,
        // Lifecycle fields carried from prev (Supabase / app state)
        status:           p.status           || 'active',
        de_installed_at:  p.de_installed_at  || null,
        de_installed_by:  p.de_installed_by  || null,
        created_by:       p.created_by       || null,
        created_at:       p.created_at       || null,
        updated_by:       p.updated_by       || null,
        updated_at:       p.updated_at       || null,
        note:             p.note             || null,
        // In-memory counters
        open_calls:       p.open_calls       || 0,
        total_calls:      p.total_calls      || 0,
      };
    }

    // Code is NOT in INSTALL_BASE_V2 — app-created or XLSX-imported.
    // Preserve the prev row exactly, defaulting any missing lifecycle
    // fields so downstream consumers don't see undefined.
    return {
      code: p.code, ast_id: p.ast_id, name: p.name,
      tesla: p.tesla, model: p.model, town: p.town, state: p.state,
      channel: p.channel, gradient: p.gradient, sw: p.sw,
      compressor: p.compressor, coldhead: p.coldhead,
      _alias_of: p._alias_of || null,
      status:           p.status           || 'active',
      de_installed_at:  p.de_installed_at  || null,
      de_installed_by:  p.de_installed_by  || null,
      created_by:       p.created_by       || null,
      created_at:       p.created_at       || null,
      updated_by:       p.updated_by       || null,
      updated_at:       p.updated_at       || null,
      note:             p.note             || null,
      open_calls:       p.open_calls       || 0,
      total_calls:      p.total_calls      || 0,
    };
  });
}
```

## 4. What This Patch Does NOT Change

- **`INSTALL_BASE_V2` array** — untouched. Hardcoded 25 rows remain canonical.
- **XLSX upload parser** — untouched in this overlay patch, but separately scheduled for Phase 2 rewrite to upsert-by-code (Option A — see [`v1.4.1_phase1_review.md` §6](v1.4.1_phase1_review.md)). The current bulk `delete + insert` flow at [index.html:2635](../index.html:2635) works in Phase 1 only because `asset_lifecycle` is empty. Once `asset_lifecycle` has rows referencing `config_assets` codes, the FK `on delete restrict` from migration 0003 §2 will fail the bulk delete. Do not run XLSX upload between asset_lifecycle backfill and the Phase 2 upsert deployment.
- **Supabase load query** — untouched (`config_assets.select('*')` continues unchanged).
- **Function call sites** — both call sites at lines 1520 and 8202 keep the same signature; the function still mutates the global `CONFIG_ASSETS`.
- **Render functions** — `_aggregateContracts()`, `renderPM()`, `renderRenewals()` see no API change; they iterate `CONFIG_ASSETS` and read the same fields they read today plus the new lifecycle fields.

## 5. Load-Order Issue Explained

```
1. Cold start
   └── applyInstallBaseOverlay()                  // CONFIG_ASSETS = [] → V2 rows
       └── CONFIG_ASSETS = 25 V2 rows, all status:'active' (defaulted)
2. Supabase response
   └── CONFIG_ASSETS = assets.map(...)             // Replaces with DB rows
       └── CONFIG_ASSETS = N DB rows (may be > 25 if app-created assets exist)
3. applyInstallBaseOverlay() called again         // ★ THE BUG ★
   └── (current behavior) rebuilds from V2,
       dropping any DB row whose code ∉ V2
       and erasing status / de_installed_at etc.
   └── (proposed behavior) merges V2 + DB,
       preserving DB-only codes and lifecycle fields
```

The patch fixes step 3 only. Steps 1 and 2 are untouched.

**Subtle correctness check:** in the proposed code, when a code appears in V2 and also in DB, V2 wins for physical attributes and DB wins for lifecycle. This is the intended product policy: V2 is the manufacturer/customer canonical source; the DB is the operational state. If a future XLSX upload changes a customer's town (e.g. relocation), the V2 hardcoded value will silently overwrite it on next reload. This is a known wart and is the reason Phase 2 should remove the V2 hardcoded fallback once the DB is the single source.

## 6. Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| App-created asset's `created_at` typed as JS Date but DB returns ISO string — type drift | Medium | Low | The patch passes through `p.created_at` as-is. Downstream consumers should call `parseFlexDate()` if they need a Date object. |
| `_alias_of` for V2 codes was always `null`; preserving prev's `_alias_of` for V2 codes could surprise callers | Low | Low | The patch hardcodes `_alias_of: null` for V2-matched rows (matching current behavior). Only DB-only rows can have a non-null `_alias_of`. |
| Schema not yet applied — `status` field doesn't exist server-side | High pre-DDL | Low | The patch defaults `status` to `'active'` if missing. Safe to ship before or after the DDL. |
| `open_calls` / `total_calls` are recomputed by ticket aggregation post-load — overlay carries them forward but they get overwritten anyway | High | None | No behavior change; same as today. |
| New asset created via app, code not in V2, then admin re-uploads XLSX without that code → bulk delete wipes the app-created asset | High when XLSX is re-uploaded | Medium | Document the XLSX bulk delete behavior. Phase 2 should change XLSX upload to upsert-only or service_role-only with explicit pre/post diff. |
| Performance — Set construction of all codes is O(N+M); fine for N=25, M ≤ ~100 | Low | None | – |

## 7. How to Verify on Staging (After DDL + Overlay Patch Both Applied)

1. **Pre-test:** confirm `CONFIG_ASSETS.length === 25` and `_aggregateContracts().rows.length === 25` (baseline).
2. **Insert synthetic asset via SQL editor:**
   ```sql
   insert into public.config_assets(code, name, model, town, state, status)
   values ('TEST-IB-A1', 'Test A1 Hospital', 'Philips 1.5T Achieva', 'Pune', 'Maharashtra', 'active');
   ```
3. **Reload app.** `CONFIG_ASSETS.length === 26` ✓. `CONFIG_ASSETS.find(a => a.code === 'TEST-IB-A1')` returns the row with `status: 'active'`.
4. **De-install via SQL:**
   ```sql
   update public.config_assets
     set status = 'de_installed',
         de_installed_at = now(),
         note = 'staging test'
     where code = 'TEST-IB-A1';
   ```
5. **Reload app.** Same row, `status: 'de_installed'`, `de_installed_at` populated, `note: 'staging test'` ✓.
6. **Without the patch** (regression check): step 5 would show `status: 'active'` and `de_installed_at: null` because the overlay rebuilt from V2.
