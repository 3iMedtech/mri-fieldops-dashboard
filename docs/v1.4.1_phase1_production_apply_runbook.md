## 10. Risks (production-specific additions to staging risks)

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Wrong project — DDL applied to staging, sandbox, or another project instead of production | Low | Critical | §1.1 visual confirmation. Always check the project pill before clicking Run. |
| Production user audit reveals users that should NOT be granted any role (e.g., test signups) | Medium | Medium | §2.3 explicitly excludes them from the seed. Step 3 of the seed inserts viewer rows for everyone in `auth.users` — this is the deliberate Phase 1 design (Phase 2 may add finer-grained gating). If you want certain users to have NO role row at all, modify Step 3 to add a `where u.email not in (...)` exclusion list. |
| Production seed inserts wrong admin email | Medium | Critical | §2.3 approval gate. Sanity check at end of Step C aborts if zero admins. |
| Production has more `auth.users` than the operator expected (e.g. dormant signups) | Medium | Low | Step 3 of seed defaults everyone to viewer. Inactive users get viewer rows but cannot write at the DB layer regardless. |
| Recursion fix not present (someone is running an older 0003) | Very Low | High | §1.3 file SHA check + §4 Verify A2 + A7 confirm `_other_active_admin_exists` + the policy referencing it. |
| Step A partial failure | Low | Medium | DDL is idempotent (`if not exists`); re-run after fixing root cause. |
| Step C seed runs without service_role | Low | Critical | Use default SQL Editor session (postgres). Sanity check aborts if zero admins. |
| Production app code path expects `_userRole` from `app_metadata.role` (Phase 1.5 gap) | Medium | Low | Same gap as staging. App still works for Admin (whose `app_meta_role='admin'`); Manager UX preserved by client-side allowlist. Phase 2 closes the gap. |
| Existing prod admin user differs from staging admin user | Low | Medium | §2.3 approval gate must be done explicitly per production user list. Do not assume staging seed list applies. |
