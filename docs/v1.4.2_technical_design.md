## 3. Track A — Renew Asset RPC + UI

### 3.1 Goals

- Admin and Manager can renew an asset's contract from the Install Base UI.
- The Renew action is atomic: prior active `asset_lifecycle` row is superseded, new active row inserted, history row appended, all in one transaction.
- Engineer/Viewer cannot Renew (UI + DB both reject).
- De-installed assets cannot be renewed via this path (`L-RTI-004`-style guard).
- All Renew events appear in the existing read-only History viewer with `event='renewed'` (already styled per `_HIST_EVENT_STYLE` in `index.html:6333`).

### 3.2 DB layer — `app_renew_asset` RPC (migration 0006)

**Signature:**
```
public.app_renew_asset(
  p_asset_code      text,
  p_contract_type   text,
  p_contract_start  date,
  p_contract_end    date,
  p_source_ref      text default null,
  p_source_customer text default null,
  p_note            text default null
) returns json
language plpgsql security definer set search_path = public
```

**Behavior summary:**
1. Captures `auth.uid()` + `auth.jwt() ->> 'email'` for audit.
2. `public.app_can_write()` — admin OR manager; viewer rejected with `42501`.
3. Validates inputs: code non-empty, `contract_type` ∈ {warranty, extended_warranty, cmc, labour_contract}, both dates present, end ≥ start.
4. `select … for update` on `config_assets` row by `upper(trim(p_asset_code))`; rejects missing (`P0002`); rejects `status='de_installed'` (`22023`).
5. `select … for update` on existing `asset_lifecycle` row with `status='active'`; takes a snapshot.
6. If a prior active row exists: `update … set status='superseded', updated_by/at` (frees the partial unique index slot).
7. `insert into asset_lifecycle (...) values (..., status='active') returning id`.
8. `insert into asset_lifecycle_history (event='renewed', before_state, after_state, actor, actor_email, source='app:renew_asset_rpc', note)`.
9. Returns `{success: true, asset_code, new_lifecycle_id, superseded_lifecycle_id, contract_type, pm_required, contract_start, contract_end, renewed_at, actor, actor_email}`.

**Why SECURITY DEFINER + locked search_path:**
- Same pattern as Phase 1 helpers (`L-SQL-001`).
- The function owner has direct INSERT/UPDATE privileges on `asset_lifecycle` and `asset_lifecycle_history`; this bypasses RLS evaluation inside the function body and avoids re-entry into `app_can_write()` for each statement. The single `app_can_write()` check at the top is the only role gate.
- Locking `search_path` prevents shim-injection attacks.

**Why NOT touch `cmc_contracts`:**
- `asset_lifecycle` is the canonical forward-looking contract registry (`source_ref` column stores the trace to `cmc_contracts.sn` if applicable).
- Touching `cmc_contracts` in the same RPC would couple Renew to the 0007 UNIQUE constraint — they should remain independent.
- Operators who want to also reflect a new contract row in `cmc_contracts` can either (a) re-run the XLSX upload (post-Track-B), or (b) write to `cmc_contracts` manually via SQL Editor.

### 3.3 Runtime layer — UI design

**New element 1 — IB row Renew button** (in the existing `.mgr-plus` Actions cell next to Edit + De-install):

```html
<button class="btn btn-ghost mgr-plus" title="Renew AN0xx" onclick="openRenewAssetModal('AN0xx')">
  <i data-lucide="rotate-cw"></i>
</button>
```

CSS visibility: existing `.mgr-plus` rule already hides for viewer-mode.

**New element 2 — Renew Asset modal** (mirror of Add Asset / Edit Asset / De-install modals):

| Field | Type | Validation |
|---|---|---|
| Asset (read-only display) | string | shown as `AN0xx — Customer Name`; sourced from `CONFIG_ASSETS` |
| Contract type | select | required; options: Warranty / Extended Warranty / CMC / Labour Contract |
| Contract start date | date input | required; `<=` end |
| Contract end date | date input | required; `>=` start |
| Source ref (sn) | text input | optional; pre-fills with prior active lifecycle's `source_ref` if any |
| Source customer | text input | optional; pre-fills with asset.name |
| Note | textarea | optional |

Submit button starts disabled; enabled when all required fields pass synchronous validation.

**New handler — `openRenewAssetModal(code)`:**
- Defensive `canManagePM()` check (matches `L-RTI-004` for de-install).
- Look up asset in `CONFIG_ASSETS`; reject if not found or status='de_installed' (matches B5.5 de-install guard).
- Optionally read the latest `asset_lifecycle` row via `_sb.from('asset_lifecycle').select('*').eq('asset_code', code).eq('status','active').single()` to pre-fill `source_ref`, `source_customer`, prior `contract_type`/`contract_end` defaults.
- Open the modal.

**New handler — `submitRenewAsset()`:**
- Re-validate inputs client-side.
- Call `_sb.rpc('app_renew_asset', { p_asset_code, p_contract_type, p_contract_start, p_contract_end, p_source_ref, p_source_customer, p_note })`.
- On success: toast/log `Renewed AN0xx`, close modal, `loadAllFromDB({ silent: true })`, refresh IB / Dashboard renewals tile / Contracts.
- On error: surface the PostgREST error message in the modal's error banner; do NOT close the modal; do NOT clear form fields. Standard error categories:
  - `42501` → "This action requires admin or manager access."
  - `P0002` → "Asset code not found."
  - `22023` → render the message verbatim ("contract end must be on or after contract start" / "asset is de_installed; cannot renew" / etc.).
  - Anything else → "Renew failed: <message>. Contact admin."

### 3.4 Audit logging

The RPC's `asset_lifecycle_history` insert is the canonical audit. In addition, the runtime handler should call `logAudit('renew_asset', {asset_code, contract_type, contract_start, contract_end})` after RPC success to maintain parity with Edit / De-install which both fire `logAudit`. Failure of `logAudit` is best-effort and does not roll back the renewal (matches existing pattern).

### 3.5 Role gating (three-layer defense, `L-RTI-009`)

| Layer | Rule | Where |
|---|---|---|
| CSS | `.mgr-plus` button hidden for viewer-mode | existing CSS rule |
| JS | `canManagePM()` defensive check in `openRenewAssetModal` + `submitRenewAsset` | new handlers |
| DB | `app_can_write()` check inside RPC | migration 0006 |

Engineer/Viewer hitting `_sb.rpc('app_renew_asset', …)` via DevTools is rejected at the DB layer with a clear 42501 error.

### 3.6 Rollback plan for Track A

| Failure mode | Default path | Notes |
|---|---|---|
| RPC has a logic bug post-apply | Apply `0006_…_ROLLBACK_…` (drops the function). Then `git revert` the Track A runtime PR. | Pre-existing `asset_lifecycle` rows remain intact; only the RPC mechanism is removed. |
| Renew button appears but errors on click | Likely RPC apply order issue (runtime deployed before SQL). Apply 0006; retry. Otherwise revert runtime. | Pre-tag smoke catches this. |
| Partial state (lifecycle row inserted, history insert failed) | Cannot occur: the entire RPC runs in one transaction. If the history insert errors, the entire RPC aborts. | Postgres transactional atomicity. |
| Future inability to rename the function signature | Drop + recreate via ROLLBACK + new migration. | The 7-arg signature should remain stable. |

## 4. Track B — `cmc_contracts` UNIQUE + safe-upsert

### 4.1 Goals

- Add a `UNIQUE` constraint on `public.cmc_contracts(sn)` so PostgREST `upsert(rows, { onConflict: 'sn' })` works.
- Replace the unsafe `delete().neq('sn', -99999) + insert()` pattern in `applyUploadedData` (`index.html` ~line 2910) with the safe-upsert pattern.
- Preserve non-XLSX rows in `cmc_contracts` (operator-edited rows, Renew-RPC trace rows, manual fix rows).

### 4.2 DB layer — UNIQUE constraint (migration 0007)

Pre-flight duplicate-sn audit (operator pastes the query result; STOP if non-zero):
```sql
select sn, count(*) as cnt
  from public.cmc_contracts
 group by sn
having count(*) > 1
 order by sn;
```

Apply:
```sql
do $$
begin
  if not exists (
    select 1 from pg_constraint
     where conrelid = 'public.cmc_contracts'::regclass
       and conname  = 'cmc_contracts_sn_unique'
  ) then
    alter table public.cmc_contracts
      add constraint cmc_contracts_sn_unique unique (sn);
  end if;
end $$;
```

Verification is built into the migration (raises exception on missing constraint post-apply).

### 4.3 Runtime layer — safe-upsert

Current code at `index.html` ~line 2910 (the unsafe pattern):
```js
await _sb.from('cmc_contracts').delete().neq('sn', -99999);
const { error } = await _sb.from('cmc_contracts').insert(cmcRows);
```

Replacement (the safe pattern, mirrors `xlsxOwnedConfigCols` from B4-partial):
```js
// Explicit whitelist of XLSX-owned columns. Any future runtime-only
// column (e.g. an Admin-edit flag, a Renew-RPC-set source) must NOT
// appear in this list, so XLSX uploads cannot blow it away.
const cmcOwnedXlsxCols = ['sn','customer','town','model','contract','start','end'];

// Project each parsed XLSX row down to the whitelist, preserving null
// for missing keys, before sending to the DB. Same pattern as
// xlsxOwnedConfigCols in B4-partial (L-RTI-005).
const cmcRowsSafe = cmcRows.map(row => {
  const out = {};
  for (const k of cmcOwnedXlsxCols) out[k] = row[k] ?? null;
  return out;
});

const { error: e } = await _sb.from('cmc_contracts').upsert(cmcRowsSafe, { onConflict: 'sn' });
if (e) {
  console.error('cmc_contracts upsert failed:', e);
  // surface to UI via the existing applyUploadedData error path
  return;
}
```

**Why this works:** PostgREST `upsert(..., { onConflict: 'sn' })` issues an `INSERT … ON CONFLICT (sn) DO UPDATE SET col1 = EXCLUDED.col1, …` for each row.  Columns NOT in the input (i.e., not in the whitelist) are not touched.  Rows whose `sn` is NOT in the XLSX upload are NOT touched.  The 0007 UNIQUE constraint is the only DB-side prerequisite.

### 4.4 Rollback plan for Track B

| Failure mode | Default path | Notes |
|---|---|---|
| Pre-flight finds duplicate sn rows | STOP. Reconcile manually before apply. | Pre-flight catches; no apply runs. |
| Constraint apply fails (some edge-case data drift after pre-flight) | Atomic — no partial state. Rerun pre-flight; reconcile; retry. | `add constraint` is atomic. |
| Safe-upsert runtime regresses an Admin XLSX upload | `git revert` the Track B runtime PR (reverts to unsafe delete+insert; admin-only gate still holds via Track C-pending). | Pre-tag smoke catches via TEST-CMC-AAA. |
| Constraint causes legitimate duplicate to fail later | Apply `0007_…_ROLLBACK_…` to drop the constraint. | Operator decision; rare. |

## 7. Track F — PM Reminder Email format + MATCH column role visibility

### 7.1 Current-state inspection

**Compose mechanism** (`composePMEmail(pmId)` at `index.html:5955-5989`):
- Uses `mailto:` URL (`window.open(mailtoUrl, '_blank')`).
- Body is a JS template literal with real `\n` newlines, wrapped in `encodeURIComponent(…)` before going into `&body=`.
- Subject format: `[PM ${urgency}] ${pm.customer} — ${pm.model} (${pm.town}) — Due ${pm.nextPMS||'TBD'}`.
- CC list is the fixed `PM_REMINDER_CC` array (4 internal emails).
- To list is dynamically computed from `getEngineersForRegion(pm.region)`.

**Why the email shows as one unformatted paragraph:** the `encodeURIComponent` of a `\n` becomes `%0A` in the URL. Some mail clients (notably Zoho when invoked via `mailto:`) interpret `%0A`-only as a soft wrap rather than a hard line break and render the body as a single paragraph. The canonical fix is to use **CRLF** (`\r\n` → `%0D%0A`), which mail clients reliably interpret as hard line breaks.

**MATCH column** (`index.html:5665`):
```js
{ key:'match', label:'MATCH', accessor: r => ({exact:0,fuzzy:1,town_model:2,none:3}[r.tier] ?? 9) }
```
This is a column in the PM Schedules table that exposes internal matching-tier classification ("exact" / "fuzzy" / "town+model" / "none"). It is a debug/audit aid for Admin/Superadmin and should not be visible to Manager or Engineer.

### 7.2 Goals (Track F)

1. Email body opens in Zoho (or any mail client) with proper line breaks, professional tone, all dynamic values populated.
2. MATCH column visible only to Admin/Superadmin.

### 7.3 Email body redesign

**Subject (unchanged structure, but use uppercase status for clarity):**
```
[PM <STATUS>] <Customer> – <Model> (<Location>) – Due <Due Date>
```
Where `<STATUS>` = `OVERDUE` if `st.code==='overdue'`, `DUE SOON` if `st.code` is in {`window_open`, `urgent`}, otherwise the existing status code uppercased.

**Body template:**
```
Dear Team,

This is a reminder that the Preventive Maintenance Service is overdue for the following machine:

Customer: <pm.customer>
Location: <pm.town>
Model: <pm.model>
Contract Type: <pm.contract>
PM Due Date: <dueDate>
PM Window: <windowStart> to <windowEnd>
Status: <st.label>
Last PM Done: <pm.lastPMS || 'N/A'>
PM Frequency: <pm.freq>x per year

Please schedule and complete the PM within the window. Once completed, kindly confirm by replying to this email with the PM report.

Regards,
Service Desk
```

The "overdue" word in the opening line should adapt to status: "overdue" if status code is overdue, "due" otherwise.

**Encoding fix:** before the `encodeURIComponent`, replace `\n` with `\r\n`:
```js
const bodyRaw = `Dear Team,\r\n\r\nThis is a reminder ...`;
// or:
const bodyRaw = template.replace(/\n/g, '\r\n');
const body = encodeURIComponent(bodyRaw);
```

This yields `%0D%0A` between lines, which Zoho and other mail clients interpret reliably as hard breaks.

### 7.4 MATCH column role visibility

The existing CSS at `index.html:436`:
```css

body:not(.superadmin-mode) .superadmin-only { display: none !important; }
```

Track F should add the `superadmin-only` class to BOTH the PM table MATCH column header (`<th>`) and the per-row MATCH cell (`<td>`). This hides the column for any role that is not Admin/Superadmin (Manager and Engineer both lose visibility).

**Implementation sketch:**
1. In `_PM_SORT_COLS` (`index.html:5653`), add a `thAttrs: 'class="superadmin-only"'` (or whatever the existing sortable-column rendering uses for arbitrary header attributes) to the MATCH entry.
2. In the PM row rendering (around `index.html:5746` where `PM_TIER_PILL_LABEL` is used), wrap the per-row MATCH cell with `class="superadmin-only"`.
3. Update the row colspan in any empty-state placeholder accordingly (matches the B5.6 column-simplification pattern).

### 7.5 Acceptance criteria (Track F)
